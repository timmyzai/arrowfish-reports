业务影响 — 服务器安全漏洞（严重遗留风险）

审计发现：测试服务器和生产服务器的Nginx均未配置默认拦截规则。当请求使用未注册的域名访问服务器时，Nginx不会拒绝请求，而是将其转发至后端应用程序。这与此前遭受木马注入攻击所利用的攻击路径完全一致。
此外，测试服务器和生产服务器共用同一个Azure网络安全组，且HTTP（80端口）和HTTPS（443端口）均对全球公网开放。攻击者可使用任意域名发送请求，直接穿透Nginx到达后端。
攻击路径后果
攻击者用任意域名发请求 → Nginx无拦截 → 直达后端 此前木马注入攻击的入口 — 攻击者通过未注册域名绕过正常路由，注入恶意代码
扫描器/爬虫探测服务器IP 服务器返回404页面和版本信息，暴露技术栈（Nginx版本、后端框架、API格式）
域名配置出错时无明确提示 服务器返回误导性错误（"用户未登录"），排查方向错误，延长故障修复时间
发现当天已全部修复（两台服务器）：
措施效果
配置Nginx默认拦截规则（返回444：静默断开连接） 未注册域名的请求被立即丢弃，零信息泄露 — 攻击者看到的只是"连接被拒绝"
HTTP和HTTPS端口均已覆盖 所有未注册域名在Nginx层即被拦截，不再转发至后端应用
已验证：正常域名访问不受影响 所有已注册的业务域名（客户端、管理后台、节点API）均正常工作
此修复关闭了木马注入攻击的入口。修改为运维级别操作，无需代码部署，零成本，零停机时间。建议后续将测试服务器和生产服务器的网络安全组分离，进一步收紧测试服务器的公网访问权限。

平台承诺"不记录用户活动"。本周审计发现前团队在多处违反了无日志政策，已当周全部修复。
违规位置前团队遗留的问题修复内容状态
安卓 MmkvManager.kt 调试日志记录完整 VPN 服务器配置、连接密钥、订阅 URL（5处日志语句） 全部替换为无敏感信息的等效日志（仅记录数量，不记录内容） 已修复
后端客户端 API 追踪中间件 启用时记录用户 IP 和完整请求体，违反无日志政策 用合规请求日志（仅方法、路径、状态码、耗时，无用户数据）替换追踪中间件 已修复
后端 middleware.Logger() 旧版本将所有请求写入磁盘 — 生成 22 GB PM2 日志文件 替换为空操作日志器；生产日志文件已手动清空（磁盘 50%→33%） 已修复，待部署生效
